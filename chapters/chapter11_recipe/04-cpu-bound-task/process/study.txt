자식 프로세스도 하나의 독립적인 '프로그램'이기 때문에 어떤 코드를 실행해야 할지 알려주기 위해 스크립트 파일 경로가 반드시 필요합니다.

fork(modulePath): 오직 새로운 Node.js 프로세스를 실행하기 위해 만들어진 함수입니다.
modulePath(파일 경로)에는 자식 프로세스가 실행할 Node.js 스크립트 파일(.js 또는 .mjs)의 경로를 전달합니다.
특징: fork는 부모와 자식 프로세스 간에 메시지를 쉽게 주고받을 수 있는 통신 채널을 자동으로 만들어 줍니다. 코드의 worker.once('message', ...) 부분이 바로 이 통신 채널을 사용해 자식 프로세스로부터 "저 준비됐어요!" 라는 메시지를 받는 부분입니다.


  ㅇ child_process worker

  'message' 이벤트는 자식 프로세스가 부모 프로세스에게 무언가 데이터를 보낼 때 발생합니다.

자식 프로세스 스크립트 안에서 process.send('보낼 메시지') 코드를 실행하면, 부모 프로세스는 'message' 이벤트를 수신하게 됩니다.

왜 이게 필요한가? (프로세스 간의 "핸드셰이크")

부모가 fork()로 자식 프로세스를 생성하는 것은 단순히 "실행시켜!"라고 명령하는 것과 같습니다. 자식 프로세스가 내부적으로 필요한 준비(예: 모듈 로딩, 초기 설정 등)를 마치는 데는 시간이 걸릴 수 있습니다.

부모는 자식이 준비가 끝났는지 확인하지 않고 바로 작업을 맡기면 안 됩니다. 그래서 다음과 같은 약속(프로토콜)을 정하는 것입니다.

부모: 자식 프로세스를 fork한다.
자식: 자신의 모든 초기화와 준비를 마친다.
자식: 준비가 끝나면 부모에게 신호를 보낸다. -> process.send('ready')
부모: 'ready' 메시지를 받으면, "아, 이제 이 자식 프로세스는 일할 준비가 되었구나"라고 인지하고 active 목록에 추가한 뒤 작업을 맡긴다.
이것이 바로 if (message === 'ready') 체크가 필요한 이유입니다.

  ㅇ cluster.fork 도 내부적으로 child_process.fork()를 호출
    cluster 모듈은 네트워크 요청을 위해 사용ㅎㅁ. 
    네트워크 포트 공유 및 로드 밸런싱: 이것이 가장 큰 차이점입니다. cluster 모듈은 여러 워커 프로세스가 하나의 네트워크 포트(예: 80번 포트)를 동시에 수신할 수 있게 해줍니다. 마스터 프로세스가 들어온 요청을 알아서 여러 워커에게 분배(로드 밸런싱)해줍니다.

  ㅇ child_process.fork()로 실행될 경우, process.on은 자식에서 부모로 데이터 전달가능한 수단! 
  BUT child_process가 nodejs program이 아닐걍우 on, send는 작동안함.
