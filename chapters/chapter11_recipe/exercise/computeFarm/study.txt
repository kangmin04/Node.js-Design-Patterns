  ㅇ HTTP VS Express
    1. HTTP
    - Node의 http 요청객체(res)는 스트림! 그래서 모든 데이터를 한번에 req.body에 담는게 아닌, 
    chunk 단위로 받음.
    - 즉, 직접 스트림을 빈 data에 넣어줘야함. 

    createServer((req, res) => {
        if (req.method === 'POST') {
            let body = '';
            req.on('data', chunk => {
                body += chunk.toString();
            });
            req.on('end', () => {
                // 직접 파싱하고 사용
                const data = JSON.parse(body);
                console.log(data.code);
                res.end('ok');
            });
        }
    }).listen(3000);

    2. EXPRESS
    - 반면 EXPRESS는 express.json() 미들웨어가 이를 해결해줌. 
    express.json()은 Content-Type 헤더가 application/json인 POST, PUT, PATCH 요청에 대해서만 위의 스트림 리스닝, 데이터 조합, 스트림 종료, JSON 파싱을 모두 해줌,

    // 2. 라우트 핸들러 등록

        // 3. req.body 사용
        // express.json() 미들웨어가 이미 파싱해서 여기에 담아주었습니다.

  ㅇ VM 
    - 안전한 코드 실행 위한 샌드박스 
    현재의 컨텍스트와는 격리된 별도의 V8 컨텍스트에서 컴파일하고 실행가능 
    - eval보다 안전. eval은 현재 스코프(실행 환경)에서 코드를 그대로 실행하여 모든 변수와 모듈(fs, process 등)에 접근할 수 있는 위험한 방식
    vm은 격리된 상태에서 코드 실행함. 외부의 변수 모듈접근이 안됨(자신만의 전역객체를 가짐) 

    메서드
    - VM.runInNewContext(code, contextObject, option)
    이 함수는 eval을 대체할 때 사용
    code: 실행할 JavaScript 코드 문자열.
    contextObject: 샌드박스 역할을 할 객체. 이 객체의 속성들이 코드 내부에서 전역 변수처럼 사용됩니다.
    options (선택): timeout 등의 옵션을 설정

    - new vm.script(code)
    코드를 미리 컴파일해줌. 그래서 자주 사용해야한다면 runInNewContext()는 매번 컴파일하기에, script로 컴파일 해두고 이후 script.runInContext 실행해줌 
     언제 사용? 동일한 코드를 여러 다른 환경(컨텍스트)에서 반복 실행할 때 성능 최적화에 유리.
     

    - vm.createContext(sandbox)
    runInNewContext는 실행마다 매번 새로운 컨텍스트 생성 
    동일한 컨텍스트에서 돌리고싶은경우 사용함. (언제 사용? 여러 스크립트를 하나의 동일한 환경에서 실행하며 상태(변수 등)를 유지해야 할 때.)
    vm.createContext(sandbox)로 재사용할 컨텍스트를 미리 만듭니다. 이 객체는 "Contextified" 상태가 됩니다.
    script.runInContext(contextifiedSandbox)를 사용해 이 컨텍스트 안에서 스크립트를 실행합니다.


    실행컨테스트 관점 ! 
    - 전역 실행 컨텍스트: 코드 실행 시 생성되는 최상위 컨텍스트이며 node에선 global 객체를 전역객체로 가짐 
    - 함수 실행 컨텍: 함수 호출될 때마다 생성됨

    모든 실행 컨텍스트는 **Lexical Environment(렉시컬 환경)**를 가지며, 이 환경은 현재 스코프 내의 변수, 함수 선언과 외부 환경(Outer Environment)에 대한 참조를 포함
    외부 환경에 대한 참조"가 바로 **스코프 체인(Scope Chain)**을 형성

    - vm.runInContext는 단순히 코드를 격리하는게 아닌 V8엔진에게 다음을 명령: 
    " 지금부터 실행할 코드를 위해, Node.js의 기본 global 객체와는 전혀 다른, 새로운 전역 환경(Global Environment)을 만들어줘. 그 새로운 전역 환경의 전역 객체(Global Object)는 내가 전달하는 sandbox 객체로 사용해."
    1. vm.runInContext(wrappedCode, sandbox)가 호출
    2. V8은 현재의 Node.js 실행 컨텍스트를 잠시 멈추고, vm을 위해 준비된 격리된 세계로 진입합니다.
    3. wrappedCode 실행위해 새로운 전역 실컨을 생성
    4. Lexical 환경 생성. 이때, 전역객체는 global이 아닌, sandbox객체. 
    스코프체인또한 최상단이 sandbox객체임. 절대로 외부의 global과 연결안됨.

  ㅇ NODE는 JIT COMPILER 방식! 
  과거의 JS는 인터프리터에 가까웠지만, 성능향상을 이유로 V8 엔진에선 Just In Time 컴파일. 
  - JIT 컴파일의 동작 원리:
    1. 초기 실행 (인터프리터): 일단 코드를 인터프리터 방식으로 빠르게 실행 (사용자가 기다리지 않도록)
    2. 모니터링 (프로파일링): 동시에, 어떤 코드가 자주 실행되는지("Hot Spot") 감시. 예를 들어, for 루프 안의 코드나 자주 호출되는 함수 등이 대상
    3. 최적화 컴파일 (컴파일러): "Hot Spot"으로 식별된 코드 조각을 가져와서 최적화된 기계어 코드로 컴파일.  이 과정에서 변수의 타입 등을 추론하여 불필요한 검사를 생략하는 등 다양한 최적화 기법이 적용
    4. 코드 교체: 다음부터 해당 코드 부분이 호출되면, 인터프리터가 해석하는 대신 미리 컴파일해둔 최적화된 기계어 코드를 직접 실행. 이로 인해 속도가 비약적으로 향상




  ㅇ THREAD 
  - Worker Thread는 메인 스레드와는 별개로 독립된 스레드에서 코드를 실행할 수 있게 해줌
   CPU 집약적인 작업을 워커 스레드에 위임하고, 메인 스레드는 본연의 역할인 다른 사용자 요청 처리를 계속

   - 부모 코드에서 
   import { Worker } from 'worker_threads';

   new Worker(filename): 새로운 워커 스레드를 생성하고, filename에 지정된 파일을 해당 스레드에서 실행합니다. 가장 기본적인 생성 방법

   worker.postMessage({ command: 'calculate', data: [1, 2, 3] }): 워커 스레드에게 데이터를(작업을) 전송. value는 구조화된 복제 알고리즘을 통해 복사되어 전달됩니다.

   worker.on('message', (result) => { console.log('결과:', result); }): 워커 스레드가 보낸 메시지를 수신하는 이벤트 리스너입니다. 작업 결과를 받을 때 주로 사용됩니다.

   worker.on('error', callback) : 워커 스레드 내부에서 처리되지 않은 예외(에러)가 발생했을 때 실행됩니다.

   worker.on('exit', callback) : 워커 스레드가 종료되었을 때 실행됩니다. 종료 코드를 인자로 받습니다.

   - 자식 코드에서 
   import { parentPort, workerData } from 'worker_threads';

   if (parentPort) { /* 워커로 실행됨 */ }: parentPort로 부모와 통신. 워커 스레드가 실행안되면 해당 값은 NULL

   parentPort.on('message', (task) => { /* 작업 수행 */ }) : 부모 스레드가 보낸 메시지를 수신하는 이벤트 리스너입니다. 작업 내용을 전달받을 때 사용됩니다.

   parentPort.postMessage(value): 부모 스레드에게 데이터를 전송합니다. 계산된 결과를 부모에게 보낼 때 사용됩니다.

   parentPort.close(): 부모와의 연결을 닫고, 워커 스레드를 종료시킵니다. worker.terminate()보다 안전한 종료 방법입니다.

ㅇ 스레드 간 데이터 공유 
        - Node.js 워커는 '메모리 격리'를 기본 원칙으로 설계됨. 각 스레드는 독립된 메모리 공간(V8 인스턴스)을 가짐.
        - 이유: '경쟁 상태(Race Condition)'를 원천적으로 방지하여 개발자가 복잡한 동기화 문제 없이 안전하게 멀티스레딩을 하도록 돕기 위함.
        - `postMessage`는 이 격리된 스레드 간에 안전하게 데이터 '복사본'을 전달하는 통신 채널 역할을 함.
        - 공유메모리 필요한 경우, sharedArrayBuffer과 Atomics로 구현.

      [심화] C/Java 방식과의 비교
       - C/Java 설계: '공유 메모리 모델'을 사용. 모든 스레드가 힙(Heap) 메모리를 공유하여, 데이터 복사 없이 주소만으로 접근하므로 성능이 극대화됨.
