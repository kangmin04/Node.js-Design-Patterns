  ㅇ EventEmitter
    - Node.js에서 **옵서버 디자인 패턴(Observer Design Pattern)**을 구현한 것입니다. 
      이 패턴은 객체 간의 느슨한 결합(loose coupling)을 가능하게 하여, 한 객체(Subject)의 상태가 변할 때 그 객체에 의존하는 다른 객체(Observers)들에게 자동으로 알림이 가도록 하는 프로그래밍 기법입니다.


    - 원리
    EventEmitter 내부는 이벤트 이름을 키(key)로, 해당 이벤트를 처리할 함수(리스너)들의 배열을 값(value)으로 가지는 객체와 같습니다.
    on('match', callback) 코드를 실행하면, 'match'라는 이름의 배열에 callback 함수가 추가됩니다.
    이렇게 특정 이벤트가 발생했을 때 실행될 함수를 미리 등록해두는 과정을 "구독(Subscription)"이라고 합니다.
    // EventEmitter 내부의 가상적인 모습
    this.events = {
        'match': [callback1, callback2, ...],
        'end': [anotherCallback, ...]
    };


    emit('match', someData) 코드가 실행되면, EventEmitter는 내부적으로 'match'라는 이름의 배열을 찾습니다.
    그리고 그 배열에 등록된 모든 리스너 함수들을 순서대로 실행합니다. 이때 emit의 두 번째 인자부터는 리스너 함수로 전달될 인자(...args)가 됩니다.
    이 과정을 "발행(Publication)"이라고 합니다.
    사용자께서 보여주신 SubsetSum 클래스를 예로 들어보겠습니다.

    class SubsetSum extends EventEmitter: SubsetSum 클래스가 EventEmitter의 모든 기능(on, emit 등)을 상속받습니다. 이제 SubsetSum의 인스턴스는 이벤트를 발행하고 구독할 수 있는 주체(Subject)가 됩니다.

    subsetSum.on('match', match => { ... }): SubsetSum 인스턴스(subsetSum)에게 'match'라는 이벤트가 발생하면 전달된 콜백 함수를 실행하라고 등록(구독)하는 부분입니다. 여기서 콜백 함수는 옵서버(Observer) 역할을 합니다.

    this.emit('match', subset): processSubset 메서드 안에서 합계가 일치하는 부분집합(subset)을 찾았을 때, emit을 통해 'match' 이벤트를 발생(발행)시킵니다. 동시에 찾은 subset 데이터를 함께 전달합니다.

    결과: 'match' 이벤트가 emit되면, on으로 등록해 두었던 콜백 함수가 실행되면서 subset 데이터를 인자로 받아 res.write(...)를 수행하게 됩니다.



  ㅇ res.write와 TCP packet(운영체제 관점)
    1. res.write() 시 데이터는 즉시 OS로 가지 않고, Node.js가 관리하는 **내부 버퍼(internal buffer)**에 먼저 기록
    2. chunk가 생길 때마다 네트워크요청(시스템콜)은 비효율적. -> 버퍼링 (내부버퍼에 어느정도 쌓일때까지 기다림)
    3. 다음의 경우 FLUSH 됨
        내부 버퍼가 가득 찼을 때
        스트림이 끝났을 때 (res.end())
        개발자가 명시적으로 버퍼를 비우라고 명령했을 때 (res.uncork() 또는 socket.flush())
        데이터 흐름이 잠시 멈춰서 "이제 보낼 때가 됐다"고 Node.js가 판단했을 때
    4. OS가 TCP 패킷 생성
        시스템콜을 통해 OS 커널에 전달. TCP 세그멘테이션. TCP완성






 ㅇ 이벤트루프 
  - Node.js 이벤트 루프의 6단계 순환
1. 타이머(Timers) 단계
역할: setTimeout()과 setInterval()로 예약된 콜백을 처리합니다.
특징: 엄밀히 말하면 "지정된 시간 후"에 실행하는 것이 아니라, **"지정된 시간이 지난 후 가장 먼저 돌아오는 타이머 단계"**에서 실행됩니다. 따라서 약간의 오차가 발생할 수 있습니다.

2. 대기 콜백(Pending Callbacks) 단계
역할: 이전 루프에서 완료되었지만 실행되지 못하고 미뤄진 I/O 콜백들을 실행합니다.
특징: 주로 TCP 오류 같은 시스템 작업의 콜백이 여기서 처리됩니다.

3. 유휴, 준비(Idle, Prepare) 단계
역할: Node.js 내부의 관리를 위해 사용되는 단계입니다.
특징: 개발자가 직접 제어할 수 없으며, 엔진이 다음 단계를 준비하는 과정입니다.

4. 폴(Poll) 단계 (가장 중요 ⭐)
역할: 새로운 I/O 이벤트를 가져오고(File 읽기, Network 요청 등) 관련 콜백을 실행합니다.
동작 방식: 1. 큐에 콜백이 있다면 다 처리할 때까지 실행합니다.
2. 큐가 비어있다면? 바로 다음 단계로 가는 게 아니라 잠시 기다립니다(Polling).
3. 만약 setImmediate가 예약되어 있다면 기다리지 않고 바로 체크(Check) 단계로 넘어갑니다.

5. 체크(Check) 단계
역할: 오직 setImmediate() 콜백만을 위한 단계입니다.
특징: Poll 단계에서 I/O 작업을 확인한 직후에 바로 실행되도록 설계되었습니다. 그래서 I/O 콜백 안에서 setImmediate를 쓰면 다음 루프가 아니라 현재 루프의 체크 단계에서 바로 실행될 확률이 높습니다.

6. 종료 콜백(Close Callbacks) 단계
역할: socket.on('close', ...)이나 db.close() 같은 연결 종료와 관련된 콜백을 처리합니다.


import { readFile } from 'node:fs/promises';

console.log('1. 시작');

async function runDemo() {
  // (A) 타이머 설정
  setTimeout(() => console.log('5. 타이머 완료 (10ms)'), 10);

  // (B) Immediate 설정
  setImmediate(() => console.log('4. Immediate 실행'));

  // (C) 비동기 파일 읽기 (I/O)
  console.log('2. 파일 읽기 요청');
  await readFile('test.txt'); // 비동기 지점
  console.log('3. 파일 읽기 완료');
}

runDemo();
console.log('끝 (동기 코드 종료)');


파일 읽기에 5ms가 걸리고, 전체 루프 한 바퀴가 매우 빠르게 돈다고 가정했을 때

1ms Microtask Queueawait 이후의 작업이 마이크로태스크 큐에 등록됨 (하지만 아직 파일 읽기가 안 끝남).
2ms Timers10ms가 안 되었으므로 통과.
3ms Poll (I/O)새로운 I/O 이벤트가 있는지 확인. (아직 파일 읽기 중)
4ms Check4. Immediate 실행 출력. (setImmediate 콜백 처리)
5ms Poll (I/O)파일 읽기 완료! OS로부터 신호를 받음. 마이크로태스크 큐에 3. 파일 읽기 완료 실행 예약.
