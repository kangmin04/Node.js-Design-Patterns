DEFINITION

1. System under test
    - sut
    - 테스트 될 대상
    - 한 컴포넌트, 모듈, 전체 코드가 될수있음 

2. Arrange, Act, Assert
    - clear, maintain하게 테스트 가능하게함
    - 모든 테스트는 다음 3단계에 따라서 작동됨
    - arrange : sut를 우한 pre condition 설정 
    - act : precondition 토대로 execute System
    - assert : expected로 실행된건지 verify

3. code coverage
    - 테스트를 실행했을 때 실제 소스 코드의 몇 %가 실행되었는지를 나타내는 지표
    - 측정 기준: 라인(Line), 분기(Branch - if/switch문), 구문(Statement), 클래스, 함수 커버리지 등
    - 주의사항: **"높은 커버리지가 높은 품질을 보장하지는 않는다"**는 점을 명심
    - 검증(Assertion) 없이 코드만 훑고 지나가는 테스트도 커버리지를 100%로 만들 수 있습니다. 이런 테스트는 아무런 가치가 없습니다.
    - 커버리지 수치 자체를 목표로 삼기보다는, 의미 있는 테스트를 작성했을 때 따라오는 부수적인 지표로 활용


4. Test doubles
    - sut를 dependencies(db, api, service)로부터 isolate 시키기 위함
    - 제한된 테스트 환경내에서 mimic real implementation (dependencies가 준비안된 상태거나 사용안될 떄에도 다른 코드를 테스트 위해서 사용)
    4.1 - Stub
            - context와 관계없이(어떤 input을 받던지) 항상 동일한 script내의 결과를 반환. 
            - 기상 api stub이면 테스트를 위해 항상 sunny만 반환
    4.2 - Spies
            - stub처럼 반환하지만 , 추가로 observer 역할
            - 메서드 얼마나 invoke됐는지, 어떤 argument받았는지, 호출 순서 등 기록함
    4.3 - Mocks 
            - Stub처럼 미리 정의된 응답을 제공하면서도, 동시에 기대되는 동작을 설정하고 이를 엄격하게 강제
            - strict validator
            - 어떤 인자 , call 횟수 등 전부 exactly defined되어있고 조금이라도 deviate되면 fail
            - 보안 프로토콜이나 API 사양 준수 여부와 같은 핵심적인 상호작용이 계약(Contract)을 따르는지 확인하는 데 유용
            - 

5. TDD
    - Test-driven development
    - 구현 코드를 작성하기 전에, 먼저 테스트를 작성하여 코드가 무엇을 해야 하는지 정의 -> 실제 요구사항에 집중할 수 있고 불필요한 코드의 유입을 방지 가능
    - RED-GREEN-REFACTOR
        - Red (실패하는 테스트 작성): 구현하고자 하는 동작을 설명하는 테스트를 먼저 작성합니다. 아직 코드가 없으므로 테스트는 실패하며, 이를 "Red" 단계라고 합니다.

        - Green (테스트 통과시키기): 테스트를 통과시킬 만큼의 최소한의 코드만 작성합니다. 이 단계의 목표는 완벽함이 아니라 기능 구현이며, 최대한 빨리 Red에서 Green으로 넘어가는 것이 핵심입니다.

        - Refactor (코드 개선): 테스트가 통과되면 코드를 정리합니다. 테스트가 계속 통과되는지 확인하면서 구조를 개선하고 중복을 제거하여 유지보수하기 좋은 코드로 만듭니다.

6. BDD
    - Behavior-driven development
    - 시스템의 동작을 기술적 지식이 있는 사람과 없는 사람 모두가 이해할 수 있는 방식으로 설명하고 검증하는 데 중점
    - 핵심은 테스트를 개발자, 제품 팀, 비즈니스 이해관계자 사이의 가교로 활용하여 요구사항에 대한 공통된 이해를 구축하고 소통을 개선
    - Gherkin 언어 사용 
        - Feature : ~ 
        - Scenerio : ~
        - Given : ~ 
        - When / And /Then
    - QA 등과 clear communication 원할 떄


7. CI
    - 지속적 통합 (Continuous Integration, CI)
    - 내가 작성한 코드가 다른 사람의 코드와 문제없이 잘 어우러지는지 '자주, 그리고 일찍' 확인
    - 자동화된 빌드와 테스트

8. CD
    - 1. 지속적 제공 (Continuous Delivery)
        - CI 파이프라인을 통과한 모든 변경 사항이 자동으로 빌드 및 테스트되어 스테이징(Staging) 또는 테스트 환경에 배포
        - 실제 운영 환경(Production)으로 출시할지 말지는 사람(개발자, QA 엔지니어 또는 릴리스 매니저)이 직접 결정 -> 최종 출시는 사람이 결정 
    - 2. 지속적 배포 (Continuous Deployment)
        - 변경 사항이 모든 자동화 테스트를 통과하면 사람의 개입 없이 즉시 운영 환경으로 배포
        - 운영 환경에 문제가 생기지 않도록 잡아낼 수 있는 회귀 테스트(Regression Test)를 포함한 강력한 자동화 테스트 세트가 필수 
        - 모든 변경 사항이 즉시 배포되기 때문에 수동 승인, 출시 전략 회의, 긴 배포 주기와 같은 병목 현상이 사라짐



9. Type of tests
    9.1 unit tests: for isolating, validating individual component
        - complete한 isolation에서 function, methods, classes 등 smallest testable units을 examine함
        - db, UI, API 등 broader context는 무시. 각 function에만 집중 
        - precision (validateCoupon()이 fail이면 coupon계산 로직 내에서 에러를 찾으면 됨 -> 디버깅 빨라짐)
        - speed (external dependencies 없기에 각 unit test는 빠른 시간 내에 종료됨)
        - unit 테스트는 잘 작동해도 다른 컴포넌트와 Integration시 에러 있을수도 있음 -> Unit 혼자 돌리진않음
    9.2 Integration test
        - components 간 잘 작동하는지 확인
        - overhead 준비필요 (실제 db, api 연동해서 확인함)
        - 각 테스트를 independent 하기 위해선, 테스트 전 known state 상태로 reset 해둬야함. (db cleaning, reinitializing external service)
        - unit test보다 slow
        - esssentail functionalities(checkout process, authentication system) 은 Integration, edge case validating은 unit test로 효율적이게 하자. 
    9.3 E2E Test
        - validate complete workflow. 
        - isolated parts만 테스트하는게 아닌, production 단계에서 모든 컴포넌트가 잘 작동하는지 확인
        - E2E는 black-box test(내부 설계를 모르는)와 유사 -> user-facing Behavior에 초점, code는 상대적으로 덜 신경씀
        - setup 복잡(db, api 등, 도커나 쿠버네티스로 편리하게 가능)
        - 작은 UI 변경에도 테스트 실패할수도있음
    
    이외의 다른 테스트들
        - performace testing(stress 상태에서 시스템 평가), usability test(user 경험 테스트), security test
        - 테스트는 check list가 아닌, spectrum임. 설계하는 시스템에 맞게, 적합한 테스트를 시행하자
    
10. Testing pyramid
    - unit, Integration, E2E 테스트를 fast, reliabel, maintainable하게 하기 위한 balance. 
    1. fast, hyper-focused한 unit test를 초기 단계에 가장 많이 실행
    2. well chosen인 E2E test 로 final safety 테스트. (가장 적게, 최소로 사용)
    -이 외에 실제 user experience를 위한 e2e를 가장 많이해라 혹은 Integration을 많이 해라 등 다양함. 



11. 실제 UNIT Test   
    11.1 equal function: 
        값 비교 후 같으면 pass -> 이후 라인이 실행됨
        다르면 마지막 인자인 콜백 메세지 출력 (어떤 err인지 state 하기위해 descriptive하게 작성하자)

        equal은 두 값이 같은 primitive values이자 same content여야만 equal이라고 함. 
        두 값이 객체라면 참조하는 메모리가 같아야만 같은 값으로 취급. 즉, 같은 구조에 같은 내용을 담아도 참조 주소가 다르면 다른 값이라고 판단
        객체의 구조, 내용만 같아도 같다고 출력하려면 deepEqual() 사용하자

        equal판단 여부 -> shallow comparison으로 === 연산자와 유사 ㅛㅗ

        일반 equal 은 shallow copy와 유사. 
        객체에 대해 shallow copy를 하면 객체의 최상위 수준의 속성만 복사 
        -> 속성값이 primitive면 값 그대로 복사되지만 객체면 메모리 참조만 복사됨 
        -> 내부 객체를 공유하게 되어 shallow copy 속성 변경 시 원본도 변경됨. 
    
    11.2 deepEqual 은 deep copy유사 
    객체 내부 모든 속성 및 중첩된 객체까지 재귀적 복사 
    -> 서로 다른 객체라 영향 안줌. 

    deepcopy 판단 알고리즘 : 
    1. objA === objB로 동일 참조인지 확인
    2. 각 obj 가 object 및 null 아닌지 확인 -> primitive 인지파악 
    3. Object.keys()로 각 객체 키 개수 파악 -> 다르면 객체 구조가다른것이기에 false
    4. 한 obj 요소로 다른 객체 순회하다가 객체 만나면 또 재귀적으로 1번부터 실행 


12. NODEJS test runner
    - Organizing test (한 파일에 관련된 code들 담아서 isolated 유지)
    - simple execution (watch mode로 테스트 recur 가능)
    - mocking too;(db, api) simulate 가능 
    - test file name convention -> filename.test.js
    - test파일 위치는 상관없음. 보통 해당 테스트 대상과 파일이 같은 디렉토리에 위치시킴
    - test 내부의 함수: function under test (FUT)
    - 한 파일에 여러개 test도 가능하다

    - test 함수에 전달되는 FUT 종류 
    테스트 함수를 정의하는 3가지 방식
        1. 동기 함수 (Synchronous Function)
        가장 기본적인 방식입니다. 함수가 실행되는 동안 에러(throw)가 발생하면 실패, 발생하지 않고 종료되면 성공으로 간주합니다.
        특징: 기다려야 할 비동기 작업이 없을 때 사용합니다.
        예시: test('테스트', t => { /* 로직 */ })

        2. 프로미스/async 함수 (Promise or async Function)
        비동기 작업을 테스트할 때 사용합니다. 프로미스가 거부(reject)되면 실패, 이행(fulfill)되면 성공으로 간주합니다.
        특징: async/await 문법을 사용할 수 있어 현대적인 자바스크립트 테스트에서 가장 권장되는 방식입니다.
        예시: test('테스트', async t => { await someAsyncWork(); })

        3. 콜백 방식 (Callback-based Function) — 상세 설명
        전통적인 비동기 처리 방식인 '콜백'을 지원하기 위한 기능입니다. 테스트 함수가 두 번째 인자로 **done**이라는 콜백 함수를 받습니다.

        A. 작동 원리
        테스트 러너는 테스트 함수에 done 인자가 정의되어 있으면, **"이 테스트는 done()이 호출될 때까지 끝난 게 아니다"**라고 판단하고 기다립니다.
        성공 케이스: done() 처럼 인자 없이 호출하거나 done(null), done(false) 등 Falsy한 값을 넘기면 성공입니다.\
        실패 케이스: done(new Error('fail')) 처럼 첫 번째 인자에 Truthy한 값(보통 에러 객체)을 넘기면 실패입니다.

        B. 주의사항: "혼용 금지" (가장 중요!)
        설명에 나와 있듯이, 콜백 방식(done)과 프로미스(async) 방식을 동시에 사용하면 안 됩니다.
        async (t, done) => { ... } 처럼 작성하면 테스트 러너는 혼란에 빠집니다. "프로미스 해결을 기다려야 하나? 아니면 done() 호출을 기다려야 하나?"를 알 수 없기 때문입니다.
        이 경우 테스트 러너는 친절하게도 "Passed a callback but also returned a Promise"라는 에러를 내뱉으며 테스트를 실패시킵니다.

13. Watch Mode
    - node --test --watch로 실행
    - test suite는 한번만 run되고, process는 stay alive ->  monitoring test files + any imported module
    - 저장되면 re-execute됨
    - context switching없고, restart안해도 됨

+ 일반적인 node --test 실행 방식
1. 테스트 명령을 내린다.
2. 새로운 Node.js 프로세스를 띄운다.
3. V8 엔진을 초기화하고, 필요한 모듈(라이브러리 등)을 메모리에 로드한다.
4. 테스트를 실행하고 프로세스를 종료한다.
5. 여기서 발생하는 오버헤드는 엄밀히 말하면 OS의 CPU 컨텍스트 스위칭보다는 **'프로세스 기동 비용(Process Startup Overhead)'**에 가깝습니다

watch mode 시 context switching없고 효율적인 이유: 
    Node.js가 테스트 실행 환경을 메모리에 일정 부분 유지하려고 시도하기 때문입니다.
    일반 모드: 테스트를 한 번 실행할 때마다 프로세스를 완전히 새로 만듭니다. (Heavy함)
    Watch 모드: * Node.js 프로세스를 죽이지 않고 살려둔 상태에서 코드 변경을 감지합니다.
    변경이 일어나면 이미 로드된 환경(V8 인스턴스 등)을 재사용하거나, 최소한의 파일만 다시 읽어 실행합니다.
    즉, **"프로세스 생성 -> 모듈 로드 -> 엔진 초기화"**라는 무거운 과정을 매번 반복하지 않으므로 전환(Switching) 비용이 획기적으로 줄어듭


14. Executing subset of all tests
    - default test(별도의 인자없이 node --test 시): 
        1. .test.js (우리가 해오던 convention)
        2. -test.js OR _test.js
        3. /test directory내의 테스트 파일들
        4. test-*.js 형식의 파일들 
        전부 node --test 시 돌아감. ! 

    - targeted test execution with custom glob
        : node --test 05-suite-test/*.test.mjs 이렇게 작성 ! (책에선 ""로 감싸는데 내가 직접해보니 "없어야 작동함)
        - 특정 feature만 돌려볼수있음
        - custom convention 가능 (꼭 test.js 형식이 아니라 spec.js 하고 argument로 경로 명시해주면 됨)


    - 테스트 컨텍스트 활용한 test 방법들
        t.skip: 
            t.skip()은 이미 작성되었지만, 특정 이유로 잠시 실행하고 싶지 않은 테스트를 건너뛸 때 사용합니다. 
            예를 들어, 버그로 인해 실패하는 테스트나 아직 고쳐지지 않은 기능을 검사하는 테스트를 임시로 비활성화하는 데 유용

        t.todo: 
            t.todo()는 아직 작성되지 않았지만, 앞으로 작성해야 할 테스트를 표시할 때 사용
            "나중에 이 테스트 만들어야지"라고 표시해두는, 말 그대로 '할 일(To-do)' 목록 같은 개념
            ***일반 test와 마찬가지로 함수 내부 코드가 실제로 실행됨! 
            test fail 일 때 failing testㄹ고 콘솔에 뜸, 다만 실제 집계는 todo 1로 잡히고 pass/fail엔 표시안됨


        t.only:
            execlusive test용

16. assert 
    - assert 모듈은 어떤 값이 참일것이라고 단언하고 검증하는데 사용 
    - assert한 내ㅐ용이 거짓이면, AssertionError로 테스트 fail 시킴
    - assert.ok(value, [message]): value가 truthy인지 확인 (true, 1, 'hello', {}, [] 등 (false, 0, '', null, undefined, NaN이 아닌 모든 값))
    - assert.strictEqual(actual, expected, [message]): 두 값이 같은지 비교 (===), (객체는 메모리 참조 주소까지 같아야함)
    -  assert.deepstrictEqual(actual, expected, [message]): 두 값이 같은지 객체, 배열 내용을 재귀적으로 비교 

    - **이때 assert의 메세지는 test가 fail 일떄만, Assertionerr로 나옴!, pass면 안나옴.
