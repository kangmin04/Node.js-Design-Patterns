1. 확장성의 세 가지 차원 (The Scale Cube)
① X축 확장: 복제 (Cloning)
    - (X축 확장은 가장 흔하고 구현하기 쉬운 방법) 동일한 애플리케이션 인스턴스를 여러 개 실행.
    - 방법: 애플리케이션의 복사본을 여러 서버(또는 프로세스)에 띄우고, 그 앞에 로드 밸런서(Load Balancer)를 두어 트래픽을 분산시킴
    - 특징: 개발 비용이 적고 즉각적인 효과를 볼 수 있습니다. 만약 인스턴스가 4개라면, 각각 전체 트래픽의 약 25%씩을 담당하게됨. 
    - 한계: 데이터베이스 자체가 병목이 되거나, 애플리케이션의 복잡도가 너무 높을 때는 이 방법만으로 해결되지 않습니다.

② Y축 확장: 서비스/기능 분해 (Decomposing)
    - Y축 확장은 애플리케이션을 기능이나 역할에 따라 여러 개의 독립적인 서비스로 쪼개는 것
    - 방법: 하나의 거대한 앱을 '사용자 인증 서비스', '결제 서비스', '상품 관리 서비스' 등으로 분리합니다. 이것이 흔히 말하는 **마이크로서비스 아키텍처(MSA)**의 핵심입니다.
    - 특징: 각 서비스는 독립적으로 개발, 배포, 확장이 가능합니다. 특정 기능(예: 결제)에만 트래픽이 몰릴 경우 해당 서비스만 별도로 확장할 수 있어 효율적입니다.  
    - 효과: 코드의 복잡성을 줄이고, 여러 팀이 동시에 다른 서비스를 개발할 수 있어 조직의 확장성도 높아집니다.

③ Z축 확장: 데이터 파티셔닝 (Data Partitioning)
    - Z축 확장은 인스턴스마다 담당하는 데이터의 영역을 나누는 것입니다.
    - 방법: 모든 인스턴스가 모든 데이터를 다루는 것이 아니라, 특정 기준(사용자의 국가, 이름 알파벳 순, ID의 해시값 등)에 따라 요청을 특정 인스턴스로 보냅니다.
    - 특징: 주로 데이터베이스 레벨에서 샤딩(Sharding)이라는 이름으로 많이 쓰이지만, 애플리케이션 레벨에서도 적용 가능
    - 효과: 데이터 규모가 너무 커서 하나의 인스턴스가 전체를 감당할 수 없을 때 매우 강력합니다. 대규모 글로벌 IT 기업들이 주로 사용하는 고급 전략입니다.


2. 수직적 확장(Vertical) vs 수중적 확장(Horizontal)
 - 수직: HW적으로 확장
 - 수평: SW적 확장

3. Node.js와 확장성
 - Node.js는 확장성에 있어 독특한 철학을 가지고 있습니다.
 - 싱글 스레드의 특성: Node.js의 자바스크립트 실행은 싱글 스레드입니다. 따라서 최신 멀티코어 CPU의 성능을 100% 활용하려면 기본적으로 **여러 개의 프로세스를 실행(X축 확장)**해야 합니다.
 - 강제적인 좋은 습관: Node.js는 처음부터 "어차피 여러 인스턴스를 돌려야 한다"는 전제를 깔고 가기 때문에, 개발자는 자연스럽게 무상태(Stateless) 설계를 지향하게 됩니다.
 - 예시 (세션 관리): 서버 메모리에 세션을 저장하면, 다른 인스턴스로 접속한 사용자는 로그아웃된 것처럼 보입니다. 이를 방지하기 위해 Node.js 개발자들은 초기부터 Redis 같은 외부 저장소에 세션을 저장하는 방식을 선호하게 됩니다.


4. CLUSTER모듈
 - Node.js는 기본적으로 싱글 스레드로 작동하므로, 멀티코어 시스템의 성능을 모두 쓰려면 코어 수만큼 프로세스를 띄워야 합니다. 
 - 이를 자동화해주는 것이 cluster 모듈입니다.
 - 하나의 **기본 프로세스(Primary Process)**가 여러 개의 **워커 프로세스(Worker Process)**를 생성(fork)합니다.
 - Primary Process: 워커들을 관리하고 외부에서 들어오는 네트워크 연결을 수신하여 워커들에게 배분
 - Worker Process: 실제 애플리케이션 로직을 실행하고 요청을 처리합니다. 각 워커는 독립된 프로세스이므로 자신만의 메모리 영역을 가짐
 - 로드 밸런싱 알고리즘: 라운드 로빈 (Round-Robin)
 - 일반적으로 한 포트(예: 8080)는 하나의 프로세스만 점유할 수 있습니다. 하지만 클러스터 모듈에서는 여러 워커가 동일한 포트를 리스닝하는 것처럼 보입니다. 이는 워커가 직접 리스닝하는 것이 아니라 Primary 프로세스에 위임하기 때문입니다.
   워커가 listen()을 호출하면 실제로는 Primary에게 "포트를 열어달라"고 요청합니다.
   Primary가 해당 포트를 잡고 있다가, 연결이 들어오면 워커에게 전달합니다.

 - cluster은 내부적으로 child_process.fork() 사용하여 워커 프로세스 생성한다. 
 child_process는 여러 프로세스 띄우면 서로 다른 포트 사용. BUT 클러스터는 Primary 프로세스가 대표 포트 리스닝 후 워커에게 라운드 로빈방식으로 분배함. 

 - cluster은 왜 프로세스를 사용할까? 
 node의 설계 철학은 상태를 공유하지말라란것. 
 전통적 스레드 모델은 메모리 공유하며 경쟁상태나 교착상태(Dead Lock) 발생 가능 
 Node는 메모리 격리를 함. -> 멀티 스레드 사용 불가. 
 => 멀티 코어 활용을 위해 멀티프로세스를 적용. (여러개의 독립적 프로세스 실행)

즉, 워커가 독립적 프로세스면 한 worker의 문제는 격리된 메모리 공간 내애서만 발생하여, 다른 worker은 영향 안받음. 종료된 worker은 primary 프로세스에게 알리고, 대체 프로세스를 생성. 
BUT 워커가 종속적인 스레드면 한 워커에서 오류가 생기면 다른 워커 스레드들도 전부 해당 문제된 데이터를 공유하기에, 전체 시스템이 멈춤 


 - cluster.workers: Primary 프로세스에서만 접근가능함. 현재 활성화된 모든 Worker 프로세스에 대한 참조. 


    5. KILL: Signal을 전송하는 시스템 콜
        - kill <PID>: 특정 프로세스 ID(<PID>)에게 **신호(Signal)**를 보내는 Unix/Linux 명령어입니다. 
        이름과 달리 꼭 '죽이는' 것만을 의미하지는 않습니다. 
        프로세스에게 특정 이벤트를 알리는 '알림' 기능에 가깝습니다.
        
        - SIGKILL: "무조건 강제 종료해!" (거부 불가능)
          SIGTERM: "정중하게 종료해줘." (기본값, 정상 종료 시도)
          SIGUSR2: "사용자 정의 신호 2 (User-defined Signal 2)". 이 신호는 미리 정해진 기능이 없습니다. 이것이 핵심입니다. 개발자가 애플리케이션 코드 내에서 이 신호를 받았을 때 어떤 동작을 할지 직접 정의할 수 있습니다.



  ㅇ PM2
    - 프로세스 관리자. 
    - 기존 node app.mjs로 직접 실행 시, 
        1. 터미널 세션에 프로세스가 종속됨. (터미널 종료 시 서버도 종료)
        2. 에러 발생 시 서버 그대로 중단 
        3. CPU코어 여러개여도 하나의 프로세스만 사용됨. 멀티 코어활용을 위해선 직접 cluster 모듈 사용해야함. 
    - pm2 start app.mjs -i max로 실행 시 
        1. pm2가 알아서 app을 백그라운드에서 실행시켜줌
        2. pm2가 내부적으로 cluster 모듈을 사용하여 현재 device의 CPU 코어 수만큼 자동으로 프로새스 복제 및 실행함. 
        3. 각 프로세스로 들어오는 요청을 자동으로 로드밸런싱해줌. 
        4. 즉, 복잡한 cluster 모듈을 관리할 필요 없이, 순수 비즈니스 로직에만 집중하면 됨. 
        5. 워커 프로세스 죽으면 자동으로 재시작. 
        6. pm2 reload all로 서비스 중단없이 새로운 버젼으로 업데이트 가능
        7. pm2 monit, pm2 logs로 관리 가능
    - 기본(fork): pm2 start app.mjs -> 하나의 프로세스만 생성
      cluster: pm2 start app.mjs -i max -> 최대 코어 개수로 프로세스 생성
    - pm2 monit, logs 시 cpu 사용량, mem(메모리 사용량), Heap size(V8엔진이 이 프로세스 위해 확보한 전체 힙 메모리 크기), Heap Usage(현재 사용중인 힙 메모리의 비율), Event Loop Latency(이벤트 루프가 한바퀴 도는데 걸리는 시간. 얼마나 빨리 도느냐가 중요)
    - fork 시 mem 사용량은 작으나, CPU 병목에 취약하여 Event Loop Latency가 높을수도
      cluster 시 mem 높으나, Event Loop Latency가 낮음. 
      추가로 cluster의 경우, 프로세스 간 데이터 공유가 필요한 경우 Redis 등을 사용해야함. 


  ㅇ CPU, CORE, Process
  CPU: 모든 계산, 명렁어 실행이 일어나는 중앙처리장치(두뇌) 
  CORE: 실제 연산을 하는 독립된 계산 유닛. 
    1개의 CPU가 1개의 코어: 싱글코어 
    1개의 CPU가 2개의 코어: 멀티코어 
    1개의 CPU가 4개의 코어: 쿼드코어 
  PROCESS: 
  - 운영체제에서 실행 중인 프로그램의 인스턴스. 자신만의 독립된 메모리 공간을 가짐.
  **코어와 프로세스의 관계:**
    - **동시성 (Concurrency):** 1개의 코어는 여러 프로세스를 아주 짧은 시간 단위로 번갈아 실행(시분할)하여 동시에 실행되는 것처럼 보이게 할 수 있다. 즉, 환 코어는 하나의 작업만 하지만, 여러 작업을 매우 빠르게 번갈아가며 실행하기에 동시에 실행되는거처럼 보인다. (실제로 매 순간엔 오직 한 코어당 한 프로세스만 점유)
    - **병렬성 (Parallelism):** 4개의 코어는 4개의 프로세스를 물리적으로 '진짜 동시에' 실행할 수 있다.
    - `pm2 start -i max`는 CPU의 코어 개수만큼 프로세스를 생성하여 **병렬성**의 이점을 극대화하는 명령어이다.
    현대의 모든 OS는 병렬성, 동시성을 동시에 사용하여 시너지를 냄. 


ㅇ Daemon (데몬)
         - 정의: 사용자와 직접적인 상호작용 없이, 운영체제의 백그라운드에서 실행되는 프로세스.
           - 핵심 특징: 터미널 세션에 종속되지 않아, 터미널을 종료해도 프로그램은 계속 실행됨.
         - 왜 필요한가? (문제점)
           - `node server.js`로 직접 실행 시, 프로세스는 터미널의 '자식'이 됨.
           - 터미널(부모)이 종료되면, 자식인 `node` 프로세스도 함께 종료되어 서버가 중단됨.
         - 데몬의 역할: "프로세스 독립시키기"
           - 프로세스를 터미널 세션으로부터 '분리(detach)'시켜, 시스템 자체에 직접 속하게 만듦.
           - 이를 통해 컴퓨터가 켜져 있는 동안 계속해서 프로그램이 동작하도록 보장함.
         - PM2와 데몬의 관계:
           - `pm2 start` 최초 실행 시, `God Daemon`이라는 PM2 자체 데몬이 백그라운드에 생성됨.
           - 이 `God Daemon`이 우리가 실행시킨 `app.mjs` 같은 자식 프로세스들을 생성하고, 죽지 않도록 감시하고 관리함.
           - `pm2 list`, `pm2 logs` 같은 명령어는 백그라운드의 `God Daemon`에게 상태를 물어보는 통신 수단임.
         - 데몬화 과정: 
            1. 프로세스 복제
            2. 부모 프로세스 즉시 종료 
            3. 자식 프로세스는 고아가 됨. 
            4. OS는 최상위 프로세스인(PID1)에 자동으로 입양 -> 터미널과는 관계없는 독립적인 프로세스
    
 ㅇ IPC vs. Redis (프로세스 간 통신/공유)
         - 목적, 범위, 확장성에서 근본적인 차이가 있음.
         - IPC (process.on / process.send):
           - 목적: 메시지 전달 (Messaging) 및 신호(Signal) 송수신.
           - 비유: 마스터-워커 간의 1:1 내부 전화 통화.
           - 범위: 단일 서버 내의 마스터-워커 프로세스 간으로 한정.
           - 데이터: 데이터의 '복사본'이 전달됨 (메모리 직접 공유 X).
           - 확장성: 수평적 확장(서버 추가) 불가능.(IPC는 단일 서버 내에서만 작동함 -> 서비스 확장으로 서버 한대 더 추가할 경우, 해당 서버와는 IPC 통신 불가. )
           - 사용 예: PM2가 내부적으로 `reload` 명령을 워커에게 전달할 때 사용.
           ( IPC의 send: 
            1. 직렬화(워커가 변경한 데이터를 Primary Process에게 전달 시 바이너리 형태 등으로 직렬화, CPU 사용됨)
            2. 프로세스 간 통신(IPC, os 커널을 옹해 Primary Process에게 전달) 
            3. 역직렬화(받은 데이터를 다시 JS 객체로 변환. CPU 사용됨)
            4. IF 내가 생각했던거처럼, 부모가 자식에서 전체 Broadcast 한다면, 다시 데이터를 N번 직렬화, IPC N번 호출, N번 역직렬화 해야함....... -> 상당한 병목
            그래서 공유를 위해선 redis 사용하는게 합리적. 
            )
         - Redis (외부 중앙 저장소):
           - 목적: 상태 공유 (State Sharing).
           - 비유: 모든 직원이 보고 쓰는 '사무실 중앙 화이트보드'.
           - 범위: 네트워크 전체. 다른 서버의 프로세스와도 공유 가능.
           - 데이터: 모든 프로세스가 '하나의 원본'을 참조함 (진정한 상태 공유).
           - 확장성: 수평적 확장의 핵심. 서버가 늘어나도 상태를 일관되게 유지.
           - 사용 예: 사용자 세션 관리, DB 캐싱, 실시간 순위표 등.
         - 결론:
           - IPC는 PM2가 내부적으로 사용하는 '내부 통신망'.

     ㅇ JWT (JSON Web Token) 기반 인증: 완전한 무상태(Stateless) 구현
             - 정의: 사용자의 인증 정보와 권한 등을 담은 '정보 객체'를 JSON 형태로 만들어, 서버의 비밀 키로 '서명'한 암호화된 토큰.
             - 핵심 아이디어: 서버는 상태를 저장하지 않고, 클라이언트가 모든 상태 정보(토큰)를 가지고 다니게 한다.
             
             - 전통적 세션 방식 vs. JWT 방식
                 - 세션 방식 ("놀이공원 손목 티켓"): 서버가 "123번 티켓은 John"이라는 장부(상태)를 직접 보관하고 매번 확인해야 함. (메모리에 직접 저장(Stateful) 또는 외부 저장소(redis) 필요)
                 - JWT 방식 ("위조 불가능한 신분증"): 서버는 신분증(JWT)을 발급만 해주고 바로 잊어버림. 어떤 서버든 신분증 자체의 유효성(서명, 만료기간)만 검증하면 됨. (Stateless)
             
             - JWT 구조: Header.Payload.Signature
                 - Header: 토큰 타입, 서명 알고리즘 등 메타데이터.
                 - Payload: 실제 전달할 내용 (사용자 ID, 권한, 만료 시간 등).
                 - Signature: Header와 Payload를 서버의 '비밀 키(Secret Key)'로 암호화한 값. **JWT 보안의 핵심.**
                 
             - JWT 보안: 위조(Forgery) vs. 탈취(Theft)
                 1. 위조 방지 (내용 수정 불가)
                     - 공격자가 Payload의 내용을 1비트라도 수정하면 (`"role":"user"` -> `"role":"admin"`), 서버만 아는 '비밀 키'가 없으므로 올바른 서명(Signature)을 다시 만들어낼 수 없음.
                     - 서버는 토큰 수신 시, 서명을 다시 계산하여 일치 여부를 확인하므로 위조된 토큰은 즉시 거부됨.
                     
                 2. 탈취 대응 (토큰 도난)
                     - 문제점: JWT는 '소유자가 임자(Bearer Token)'이므로, 유효한 토큰이 탈취되면 공격자가 주인 행세를 할 수 있음.
                     - 대응 전략:
                         ① 짧은 만료 기간 (Short Expiry Time): Access Token의 유효 기간을 5분, 15분 등으로 매우 짧게 설정하여 탈취 시 피해 최소화. (가장 중요)
                         ② 리프레시 토큰 (Refresh Token) 패턴: 유효기간이 긴(예: 7일) Refresh Token을 별도로 발급. Access Token이 만료되면 이 Refresh Token으로 조용히 새 Access Token을 재발급받아 사용자 경험을 유지.
                         ③ HTTPS 통신 필수: 네트워크 중간에서 토큰을 엿보는 '중간자 공격(MITM)'을 원천 차단.
                         ④ (심화) 토큰 폐기 목록: 로그아웃, 비밀번호 변경등으로 더 이상 안쓰는 무효화된 토큰 목록을 Redis 등에 저장하여, 요청 시마다 해당 목록을 한번 더 확인하는 방식. (완벽한 Stateless를 약간 포기하고 보안 강화)
                         
             - 결론: JWT는 서버의 상태 저장 책임을 완전히 제거하여, 어떤 서버 인스턴스가 요청을 처리하든 일관된 인증을 제공함. 이는 로드 밸런싱, 오토 스케일링, 마이크로서비스 아키텍처 환경에서 확장성을 극대화하는 가장 선호되는 방식임.


     ㅇ Stateful vs. Stateless 서버: 분산 시스템의 핵심 개념
         - 정의: 여러 대의 서버(인스턴스)로 서비스를 확장할 때, '상태(State)'를 어디에, 어떻게 저장하고 관리할 것인가에 대한 아키텍처 구분.
         - '상태'란? 사용자의 로그인 정보, 장바구니 내용, 게임 진행 상황, 조회수 등 시간이 지나면서 변할 수 있는 모든 데이터.

         1. Stateful 서버 (상태 유지 서버)
             - 개념: 서버가 이전 요청의 결과나 클라이언트의 상태를 **자신의 메모리 안에** 직접 저장하고 기억하는 방식.
             - 코드 수준 예시:
                 // 서버 프로세스의 메모리 변수에 상태(조회수)를 저장
                 let viewCount = 0; 
                 
                 app.get('/', (req, res) => {
                   viewCount++; // 요청이 올 때마다 자신의 메모리에 있는 값을 변경
                   res.send(`조회수: ${viewCount}`);
                 });
             
             - 문제점: 수평 확장(Scale-out) 시 데이터 불일치 발생
                 - 서버 A와 B, 두 대를 운영할 경우, 각 서버의 viewCount는 별개로 존재함.
                 - 로드 밸런서가 요청을 분산시키면, 사용자는 서버 A의 viewCount(예: 5)와 서버 B의 viewCount(예: 3)를 번갈아 보게 되어 일관성이 깨짐.
                 - 서버 하나가 다운되면, 그 서버의 메모리에 있던 모든 상태(로그인 세션, 장바구니 등)가 소실됨.
             - 해결책: Sticky Session (특정 사용자를 특정 서버에 고정시키는 임시방편)

         2. Stateless 서버 (상태 비저장 서버)
             - 개념: 서버가 요청 처리에 필요한 모든 정보를 요청 자체에서 얻거나, **외부의 공유 저장소**에서 가져옴. 서버 자체는 아무것도 기억하지 않음.
             - 비유: 자판기 (누가, 언제, 무엇을 뽑았는지 전혀 기억하지 않고, 들어온 요청만 처리함)
             - 코드 수준 예시:
                 // 상태 관리를 외부 공유 저장소(예: Redis)에 위임
                 const redisClient = redis.createClient();

                 app.get('/', async (req, res) => {
                   // 상태를 읽고 쓰는 모든 작업을 외부 저장소에 요청
                   const newViewCount = await redisClient.incr('pageViews');
                   res.send(`조회수: ${newViewCount}`);
                 });

             - 장점:
                 - 완벽한 수평 확장성: 어떤 서버가 요청을 처리하든 항상 동일한 외부 저장소를 바라보므로 데이터 일관성이 보장됨. 서버를 수백 대로 늘려도 문제가 없음.
                 - 높은 안정성 및 유연성: 서버 하나가 다운되어도 상태는 외부 저장소에 안전하게 보관되어 있으므로, 다른 서버가 즉시 이어서 작업을 처리할 수 있음.
                 - 서버 배포 및 관리가 단순해짐.

         - 결론: 현대적인 대규모 웹 서비스는 안정적인 확장성과 고가용성을 위해 **Stateless 아키텍처를 지향**하는 것이 표준. 상태 관리는 Redis와 같은 고속 인메모리 DB나 영구 DB에 위임하는 것이 일반적임.




     ㅇ Socket.IO와 Sticky Load Balancing: 필수적인 조합
         - 문제점: 왜 Socket.IO는 여러 서버 환경에서 그냥 동작하지 않는가?
             - Socket.IO는 클라이언트와 '특정 서버 프로세스' 간의 지속적이고 상태 기반(Stateful)인 연결(주로 WebSocket)을 생성함.
             - 로드 밸런서가 첫 연결 요청(HTTP Handshake)은 서버 A로 보내고, 다음 통신(Polling 등)은 서버 B로 보내면, 서버 B는 해당 클라이언트에 대한 정보를 전혀 모르므로 연결이 즉시 끊김.

         - 해결 방안 1: Sticky Load Balancing (기본이자 필수)
             - 정의: 로드 밸런서가 특정 Socket.IO 세션에서 발생하는 모든 후속 요청을, 첫 연결이 수립된 동일한 서버 인스턴스로만 보내도록 '고정'하는 방식.
             - 동작: 일반적인 쿠키 기반 고정성(Cookie-based Stickiness) 기능을 사용함. 클라이언트의 첫 연결(Handshake)에 대한 응답에 로드 밸런서가 쿠키를 심어주고, 클라이언트는 이후 모든 요청에 해당 쿠키를 포함하여 항상 같은 서버로 찾아갈 수 있게 됨.
             - 한계: 이 방식만으로는 '서버 간 메시지 공유'가 불가능함. 예를 들어 서버 A에 접속한 사용자는 서버 B에서 보낸 메시지(`emit`)를 직접 받을 수 없음. 또한, 고정된 서버가 다운되면 해당 서버에 연결된 모든 클라이언트의 연결이 끊기는 단점은 여전함.

         - 해결 방안 2: Socket.IO 어댑터(Adapter) 사용 (진정한 확장성)
             - 대표 예시: `socket.io-redis` 어댑터.
             - 개념: 모든 서버 인스턴스가 중앙의 Redis Pub/Sub 기능을 구독하여, 서로 메시지를 주고받을 수 있는 통로를 만드는 방식.
             - 동작 원리:
                 1. 서버 A가 특정 방(room)에 메시지를 보낼 때 (`io.to('news-room').emit(...)`), 자신의 클라이언트에게만 보내는 것이 아니라, 연결된 Redis에 메시지를 발행(Publish)함.
                 2. Redis를 구독 중인 다른 모든 서버(B, C, ...)가 이 메시지를 수신함.
                 3. 각 서버는 메시지를 수신한 후, 자신의 서버에 연결된 클라이언트 중 'news-room'에 속한 클라이언트가 있다면 그들에게 메시지를 전달함.
             - 결과: 사용자가 어떤 서버 인스턴스에 접속해 있든, 클러스터 전체에 메시지를 브로드캐스팅할 수 있게 됨.

         - 최종 결론 (Best Practice):
             "Socket.IO를 제대로 확장하려면 Sticky Session과 Redis 어댑터, 둘 다 필요하다."
             1. **Sticky Session**: 개별 클라이언트와 서버 인스턴스 간의 '연결' 자체를 안정적으로 유지하는 역할. (연결의 안정성)
             2. **Redis Adapter**: 모든 서버 인스턴스 간에 '메시지'를 자유롭게 공유하고 전파하는 역할. (메시지의 확장성)
             => 이 두 가지를 함께 사용해야만, 안정적이면서도 수평적으로 확장이 가능한 대규모 실시간 애플리케이션을 구축할 수 있음.



     ㅇ Forward Proxy vs. Reverse Proxy: 누구를 위한 대리인인가?(Forward는 클라이언트를 위해! 클라이언트를 숨겨줌.  Reverse는 서버를 위해. 서버를 숨겨줌.)
         - 프록시(Proxy)의 공통점: 클라이언트와 서버 사이에서 요청과 응답을 중계하는 '대리인' 서버.
         - 핵심 차이점: 프록시가 '누구를 위해' 일하는가에 따라 Forward와 Reverse로 나뉨.

         1. 포워드 프록시 (Forward Proxy)
             - 개념: **클라이언트를 위한** 프록시. 클라이언트가 인터넷에 요청을 보낼 때 거쳐 가는 서버.
             - 흐름: `클라이언트 -> [포워드 프록시] -> 인터넷(외부 서버)`
             - 특징: 
                 - 요청을 받는 **최종 서버는 실제 클라이언트의 IP를 알 수 없음.** 오직 포워드 프록시 서버의 IP만 보게 됨. (클라이언트가 숨겨짐)
                 - 주로 내부망(사내 네트워크, 학교망 등)에 속한 클라이언트들이 인터넷에 접근할 때 사용됨.
             - 주요 사용 목적:
                 - **캐싱**: 자주 사용되는 콘텐츠를 프록시 서버에 캐싱하여 내부 사용자들의 접근 속도를 향상.
                 - **보안 및 접근 제어**: 내부망에서 특정 웹사이트(유해 사이트 등)에 접근하는 것을 막는 방화벽 역할.
                 - **익명성**: 클라이언트의 IP를 감추고 인터넷에 접속할 수 있음.

         2. 리버스 프록시 (Reverse Proxy)
             - 개념: **서버를 위한** 프록시. 인터넷의 불특정 다수 클라이언트로부터 오는 요청을 대신 받아 내부 서버로 전달하는 서버.
             - 흐름: `클라이언트 -> [리버스 프록시] -> 내부 서버들 (WAS)`
             - 특징:
                 - **클라이언트는 자신이 요청을 보내는 실제 서버의 IP를 알 수 없음.** 오직 리버스 프록시 서버의 IP(대표 도메인 주소)만 알게 됨. (서버가 숨겨짐)
                 - **Node.js/Express 환경에서 Nginx를 사용하는 것이 바로 이 방식.**
             - 주요 사용 목적:
                 - **로드 밸런싱**: 여러 대의 내부 서버로 요청을 균등하게 분산.
                 - **보안**: 내부 서버의 존재를 숨기고, DDoS 공격 방어, SSL 암호화(SSL Termination) 등을 전담하여 내부 서버를 보호.
                 - **정적 콘텐츠 처리**: 이미지, CSS, JS 등 정적 파일 요청을 리버스 프록시가 직접 처리하여 WAS(Node.js 서버)의 부담을 줄임.
                 - **캐싱**: 자주 요청되는 응답 결과를 캐싱하여 내부 서버로 요청이 전달되지 않게 함으로써 성능 향상.
             - cluster 모듈보다, reverse proxy가 더 선호됨! (프로세스 뿐만 아니라 다른 머신으로도 load balancing가능함. sticky load balancing도 지원 가능. most powerful한 load balancing algorithm 선택 가능)




     ㅇ Cluster 모듈 vs. 리버스 프록시: 무엇을, 언제 사용해야 하는가?
         - 공통점: 둘 다 여러 프로세스에 작업을 분산하여, 싱글 스레드인 Node.js의 한계를 극복하고 성능을 향상시키는 데 사용됨.
         - 핵심 차이: '어떤 레벨'에서 '어떻게' 분산하는가에 대한 스코프와 역할이 완전히 다름.

         1. 동작 방식 및 스코프의 차이

          | 구분 | Cluster 모듈 | 리버스 프록시 (Nginx) |
          | 목적/스코프 | 단일 서버 내의 멀티코어 CPU 활용 극대화 | 여러 서버(머신) 간의 트래픽 분산 |
          | 확장 방향 | **수직 확장 (Scale-up)**: 한 머신의 성능을 최대로 끌어냄 | **수평 확장 (Scale-out)**: 여러 머신으로 부하를 나눔 |
          | 동작 레벨 | **Node.js 프로세스 레벨**: 하나의 앱 코드를 여러 프로세스로 복제 | **네트워크 레벨**: HTTP/TCP 요청을 다른 IP/Port로 전달 |
          | 포트 관리| 하나의 포트를 여러 워커 프로세스가 공유 (Primary가 중계) | 각 앱/프로세스는 **고유한 포트**를 가져야 함 (예: 3000, 3001) |
          | 주요 기능 | - CPU 코어 수만큼 워커 생성<br>- 워커 프로세스 관리 (사망 시 자동 재생성) | - **로드 밸런싱** (다양한 알고리즘)<br>- **SSL 암호화 처리**<br>- **정적 파일 서빙**<br>- 캐싱, 보안, 압축 등 |
          | 장애 대응 | Primary 프로세스가 죽으면 전체 앱 중단 | 서버 하나가 죽어도 다른 서버로 요청을 보내 서비스 유지 가능 |

         2. 그래서 뭘 써야 하는가? (결론 및 권장 아키텍처)
             - 질문 자체가 "A vs B"가 아닙니다. 이 둘은 경쟁 관계가 아닌, **서로 다른 계층에서 협력하는 관계**입니다.
             - **현대적인 프로덕션 환경에서는 이 둘을 '함께' 사용합니다.**

         3. 최종 권장 아키텍처 (Best Practice)
             - "리버스 프록시 + (각 서버별) 클러스터 모듈"
             
             1. **1계층: 리버스 프록시 (Nginx)**
                 - 웹 서버의 '얼굴' 역할을 하며 인터넷의 모든 요청(Port 80/443)을 가장 먼저 받습니다.
                 - SSL 인증서 처리, 정적 파일(이미지, CSS) 서빙, 압축 등을 수행합니다.
                 - 로드 밸런싱 규칙에 따라 내부에서 실행 중인 여러 Node.js 서버(예: 서버1의 3000번 포트, 서버2의 3000번 포트)로 요청을 분산시킵니다.
             
             2. **2계층: Node.js 애플리케이션 (Cluster 모듈 적용)**
                 - Nginx로부터 요청을 받은 각 서버(서버1, 서버2) 내부에서는 `cluster` 모듈이 동작합니다.
                 - `cluster` 모듈은 해당 서버의 CPU 코어 수만큼 워커 프로세스를 생성하여, Nginx로부터 받은 요청을 다시 워커들에게 분배합니다.
                 - 이를 통해 단일 서버의 리소스를 낭비 없이 모두 활용할 수 있습니다.
             
             - 흐름 요약:
                 `사용자 -> Nginx -> [서버1(3000포트), 서버2(3000포트)] 중 하나로 분산 -> 선택된 서버의 Primary 프로세스가 받음 -> 내부 워커 프로세스 중 하나로 분산`

         - 결론: 리버스 프록시는 **여러 대의 서버**로 서비스를 확장하고 안정성과 다양한 부가기능을 제공하기 위해 **필수적**이며, 클러스터 모듈은 **개별 서버**의 성능을 최대한 끌어올리기 위한 **최적화** 기법입니다.

  ㅇ UNIX 파일 시스템(FHS) 
  / (루트 디렉토리)
├── bin/      # 기본적인 명령어(ls, cp 등)가 있는 곳 (모든 사용자가 사용)
├── sbin/     # 시스템 관리자용 명령어(reboot, fdisk 등)
|
├── etc/      # ★★★ 시스템 전체에 적용되는 '설정(Configuration) 파일'들이 있는 곳
|             # (예: /etc/nginx/nginx.conf)
|
├── home/     # ★★★ 일반 사용자들의 개인 홈 디렉토리. '우리 집'들이 모여있는 곳
│   └── user/ # (예: /home/user/workspace/chapters 가 여기에 있을 수 있음)
|
├── usr/      # 일반 사용자를 위한 프로그램과 데이터
│   ├── bin/  # ( /bin 에 없는 추가적인 프로그램들)
│   └── local/ # ★★★ 사용자가 직접 컴파일해서 '설치(install)'한 프로그램들이 위치하는 곳
│       ├── bin/     # (설치한 프로그램의 실행 파일)
│       └── etc/     # (설치한 프로그램의 설정 파일)
│       └── nginx/   # (Nginx를 소스 컴파일로 설치했다면 여기에 생김)
│           └── conf/
│               └── nginx.conf # ★★★
|
└── var/      # 내용이 자주 바뀌는 파일들 (로그, 캐시, 데이터베이스 파일 등)
    └── log/
        └── nginx/
            ├── access.log
            └── error.log

  - nginx의 경우, 내 프로젝트 일부에서 실행되는 라이브러리가 아닌, 시스템 전체에서 작동하는 독립적 서비스. (데몬)
  만약 내 프로젝트 app.mjs가 죽어도 nginx는 살아있어야 함 -> 내 프로젝트 내가 아닌, 시스템의 모든 사용자가 참조 가능한 공용위치에 두어야 함
